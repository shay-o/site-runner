# Three-layer Purpose / Criteria / Report stack with versioning

Reviews of the Redo.io Site are organised as three versioned, separable artifacts: a **Purpose** document (what the Site should do, jointly owned with the team), **Criteria** derived from a specific Purpose version (the review questions, human-reviewed before use), and **Reports** produced by Review Runs scored against a specific Criteria version. Reports are comparable across Runs *only* when they share a Criteria version; changing the Purpose triggers a deliberate Criteria Refresh that starts a new regression-comparison baseline.

The obvious alternative is to regenerate Criteria from the current Site description on every Run. It was rejected because score changes between Runs would conflate two different things — the Site improving and the Criteria drifting — destroying the regression value that is the whole point of running Reviews more than once.
