# site-runner — Architecture & Roadmap

## Purpose

A tool that drives a real browser through a **Journey** (a scripted path through a Site) on a fully rendered website, captures everything generated by those actions (screenshots, HTTP, WebSocket frames, console, dataLayer), and runs one of two analysis passes:

- **Usability Review Run** — score the captured artifacts against a Criteria version derived from the Site's Purpose, producing a Usability Review Report. (Capture works; analysis pass is stubbed.)
- **Analytics Data Collection Run** — decode vendor-specific tracking beacons (GA4, Meta Pixel, Streamlit/Fivetran, etc.) into normalised events and produce an Analytics Data Collection Report.

See [CONTEXT.md](CONTEXT.md) for the full domain glossary.

Use cases:
- Validate that analytics events fire correctly after a Site change.
- Audit what tracking data a Site emits for a given Journey.
- Investigate tracking gaps (steps that produce no events).
- Assess whether a Site lets users accomplish their Journeys (Usability Review).
- (Future) Regression testing against a baseline of expected events.

The browser used for each Run is ephemeral — fresh isolated context, no shared state with the user's personal browser.

---

## Target Site for v1: tool.redoio.info

The initial development target is `tool.redoio.info`, a **Streamlit-based** application. A reference HAR capture from this site has driven several architecture decisions:

**Tracking footprint:**
- No GA4, GTM, Meta Pixel, or any traditional web analytics.
- The only telemetry is Streamlit's built-in event stream sent to `webhooks.fivetran.com/webhooks/<id>` as JSON POSTs.
- Three event types observed: `viewReport` (on load), `updateReport` (on app updates), `pageProfile` (rich payload describing the entire widget tree, library attributions, browser info).

**Communication model:**
- All real app behavior flows over a single persistent WebSocket: `wss://tool.redoio.info/_stcore/stream`.
- Frames are binary protobuf (Streamlit's `ForwardMsg` schema).
- Internal page navigation does NOT make HTTP requests — it sends WebSocket frames and receives new app state in return.
- Static HTTP requests after initial load are minimal; the action surface lives in WebSocket traffic.

**Key implications driving design:**

| Finding | Architecture impact |
|---------|--------------------|
| Tracking is JSON to a Fivetran webhook | First decoder is `StreamlitFivetran`, not GA4 |
| App state lives in WebSocket frames | Runner must capture WebSocket frames alongside HTTP |
| `pageProfile` includes `"isWebdriver"` field | Runner must launch Chromium with `--disable-blink-features=AutomationControlled` to avoid flagging captures as bot traffic |
| Persistent WebSocket means `networkidle` never fires cleanly | Need `wait_for_selector` and `wait_ms` settle options, not just network idle |
| Internal nav uses WS, not HTTP | `goto` only useful for external nav; `click` handles internal nav |
| No consent banner | `consent: ignore` is a valid default for Streamlit apps |

## Component Detail

### 1. Journey YAML

Declarative user-authored description of a Journey through a Site. Each Journey lives at `sites/<site>/journeys/<slug>.yaml`; the filename stem is the stable Journey slug, validated at load time. Each step has a unique `id` used throughout the Run artifact and Report.

```yaml
journey: oaklandish-pdp-flow
role: shopper
use_case: buy-a-hoodie         # references sites/oaklandish/use-cases/buy-a-hoodie.md
start_url: https://www.oaklandish.com
viewport: { width: 1280, height: 800 }
consent:
  strategy: accept_all   # options: accept_all | pre_seed_cookies | ignore | custom
success_condition:
  description: Cart shows the selected hoodie at the right size and price.
  check:
    type: selector_visible
    selector: '[data-testid="cart-line-item"]'
steps:
  - id: load_home
    action: goto
    url: https://www.oaklandish.com
    wait_for: networkidle

  - id: open_collection
    action: click
    locator: { role: link, name: "Shop" }

  - id: open_pdp
    action: click
    locator: { role: link, name_contains: "Town Hoodie" }

  - id: select_size
    action: select
    locator: { role: combobox, name: "Size" }
    value: "L"

  - id: add_to_cart
    action: click
    locator: { role: button, name: "Add to cart" }
    settle: { network_idle_ms: 1500 }
```

**Locator strategy:** Semantic Playwright locators (`get_by_role`, `get_by_text`) rather than CSS or XPath selectors. These survive minor DOM changes and read like natural language.

**`settle`:** A configurable post-action settle window. Critical for capturing beacons that fire asynchronously after the triggering action — especially `navigator.sendBeacon` calls and GA4 batched events.

Settle strategies:
- `network_idle_ms: <N>` — wait for N ms of network idle (unreliable when WebSocket is open).
- `wait_ms: <N>` — fixed delay after action.
- `wait_for_selector: <css>` — wait for a DOM element to appear.
- `wait_for_request: <url_pattern>` — wait for a specific request to fire.

For sites with persistent WebSocket connections (Streamlit, sites with long-poll), prefer `wait_for_selector` or `wait_ms` over `network_idle_ms`.

### 2. Runner

Python + Playwright. Responsible for executing Journeys against a live browser. One Journey + one `--type` produces one Run.

**Browser isolation:**
- Fresh `BrowserContext` per Run with `record_har_path` enabled.
- No profile reuse. Storage state is ephemeral — no cookies, credentials, or history from the user's personal browser.
- Headed Chromium by default (reduces bot-detection false positives). `--headless` flag available for CI.
- Launched with `--disable-blink-features=AutomationControlled` and an init script that hides `navigator.webdriver`. This prevents sites (notably Streamlit) from flagging captures as bot traffic in their telemetry.

**Per-step execution loop:**
1. Record `step_start_ts`.
2. Execute the action (click, goto, fill, select, etc.).
3. Await the `settle` window (network idle, fixed delay, or specific request pattern).
4. Snapshot `window.dataLayer` via `page.evaluate`.
5. Capture before/after screenshots.
6. Record `step_end_ts`.

**Request tagging:**
Subscribe to `page.on('request')` in real time. Each captured request is tagged with the `step_id` that was active at request-start time. This is the join key between network events and user actions in the report.

**WebSocket capture:**
Subscribe to `page.on('websocket')` and listen to each WebSocket's `framesent` and `framereceived` events. Each frame is logged with the active `step_id`, timestamp, direction, opcode, and payload size. Binary frames have payloads stored as base64. For sites where the WebSocket carries application state (Streamlit, etc.), this is essential — and for tracking-focused sites where WebSocket is rare, the overhead is negligible.

WebSocket frames are written to `websocket_frames.jsonl` in the run artifact. Decoders may opt into WebSocket frames (e.g., a future Streamlit protobuf decoder), but by default the v1 pipeline only records and counts them.

**Note on beacon attribution:** `sendBeacon` calls and GA4 batched events often fire slightly after the action that triggered them — potentially during the *next* step's window. The settle window exists to hold attribution to the correct step. This is an inherent challenge; the report should surface ambiguous attributions.

### 3. Run Artifact

One directory per Run, under the Site's `runs/` folder. Stable layout so future tooling can parse Runs programmatically. Directory name encodes `<journey-slug>_<run-type>_<timestamp>` so `ls` groups Runs by Journey.

```
sites/redoio/runs/redoio-home_analytics_2026-05-07T14-30-12/
  journey.yaml             # Frozen copy of the Journey as executed
  run.json                 # Run metadata: journey, role, use_case, run_type, runner_version, success_condition result
  session.har              # Full HAR with embedded response bodies
  steps.jsonl              # One JSON line per step: id, action, timing, screenshot paths
  requests.jsonl           # One JSON line per HTTP request: url, method, body, step_id, timestamp
  websocket_frames.jsonl   # One JSON line per WS frame: ws_url, direction, opcode, len, data_b64, step_id, ts
  console.jsonl            # Browser console log
  datalayer.jsonl          # window.dataLayer snapshot after each step (empty on Streamlit sites)
  screenshots/
    load_home_before.png
    load_home_after.png
    ...
```

### 4. Decoder Pass

Pure functions — no LLM, no network. Each decoder is a small module implementing:

```python
class Decoder(Protocol):
    name: str
    def matches(self, request: CapturedRequest) -> bool: ...
    def decode(self, request: CapturedRequest) -> DecodedEvent: ...
```

All decoders produce a normalized `DecodedEvent`:

```python
{
  "vendor": "ga4",
  "event_name": "add_to_cart",
  "params": {
    "item_id": "SKU-123",
    "value": 65.00,
    "currency": "USD",
    ...
  },
  "client_id": "...",
  "session_id": "...",
  "step_id": "add_to_cart",       # from request tag
  "raw_url": "https://...",
  "raw_body": "..."               # preserved for debugging
}
```

**Priority vendor decoders (v1):**
| Vendor | Endpoint pattern | Notes |
|--------|-----------------|-------|
| Streamlit / Fivetran | `webhooks.fivetran.com/webhooks/<id>` | Highest priority — only tracking on `tool.redoio.info`. Clean JSON body with `event`, `anonymousId`, `sessionId`, etc. |
| GA4 | `google-analytics.com/g/collect` | Most common analytics on e-com sites |
| Shopify Trekkie | `monorail-edge.shopifysvc.com` | Shopify built-in analytics |
| Meta Pixel | `facebook.com/tr` | |
| TikTok Pixel | `analytics.tiktok.com` | |
| GTM (server-side) | Custom endpoints via GTM container ID | Pattern varies per deployment |

**Unknown beacon fallback:** Any request that no decoder matches but looks like a tracking call (small payload, no response body expected) is flagged as `vendor: "unknown"` in the output. This surfaces new/changed vendor endpoints in the report rather than silently dropping them.

The `DecodedEvent` shape is designed as the future unit of comparison for baseline regression testing.

---

### 5. Interpreter (LLM)

Runs once per step after decoding. Input per step:
- Step spec (action, locator, etc.)
- Before/after screenshots
- List of `DecodedEvent` objects for that step
- List of undecoded requests flagged for that step

Output: plain-English paragraph describing what the user action did, what tracking data was generated, and any anomalies (missing expected events, unexpected parameters, undecoded beacons).

**Provider:** OpenRouter (`https://openrouter.ai/api/v1`), model `anthropic/claude-sonnet-4`. Configurable via `OPENROUTER_MODEL` env var.

One LLM call per step (not per request) to keep cost and latency bounded.

---

### 6. Report

Single self-contained HTML file per run. Sections:
- **Run summary:** URL, timestamp, total steps, total events decoded.
- **Step-by-step timeline:** For each step — action description, before/after screenshots, decoded events table, LLM interpretation, raw requests collapsible.
- **Event index:** All decoded events across all steps, sortable by vendor / event name / step.
- **Raw HAR link:** For power users who want the full capture.

## Data Flow Summary

```
Action YAML
  → Runner executes steps, records HAR + per-request step tags
  → Decoder pass produces DecodedEvent[] per request
  → Interpreter produces per-step natural language summary
  → Report assembles everything into HTML
```

## CLI Design

```bash
# Run a Journey as an Analytics Data Collection Run (default)
site-runner run sites/redoio/journeys/bias_analysis.yaml

# Run as a Usability Review Run (analysis pass stubbed for now)
site-runner run sites/redoio/journeys/bias_analysis.yaml --type usability

# Run headless (CI mode)
site-runner run sites/redoio/journeys/bias_analysis.yaml --headless

# Save output to a specific directory
site-runner run sites/redoio/journeys/bias_analysis.yaml --out runs/my-run

# Future:
#   site-runner decode <run-dir>     # re-run decoder pass without re-driving the browser
#   site-runner report <run-dir>     # re-render report without re-running
```

## Known Risks & Mitigations

| Risk | Mitigation |
|------|-----------|
| Bot detection blocks headless browser | Use headed Chromium by default; `playwright-stealth` if needed |
| Beacon fires after action settle window | Configurable `settle` per step; report flags late-arriving beacons |
| `networkidle` never fires (WebSocket/long-poll) | `wait_for_request` escape hatch on steps |
| Vendor changes beacon endpoint or format | Unknown beacon fallback surfaces it in the report |
| Consent banner blocks all tracking | `consent` strategy in Journey YAML; `accept_all` is default |
| GA4 batches multiple events in one POST | GA4 decoder handles newline-separated event batches in body |
| Service-worker-cached responses absent from HAR | Document limitation; note in report if SW is detected |

## Technology Choices

| Concern | Choice | Rationale |
|---------|--------|-----------|
| Browser automation | Playwright (Python) | Native HAR recording, semantic locators, good Python support |
| Browser | Headed Chromium | Fewer bot-detection issues than headless |
| LLM API | OpenRouter → `anthropic/claude-sonnet-4` | Per project global preference |
| Action format | YAML | Human-readable, easy to edit, no code required |
| Output | Filesystem artifacts + HTML report | No database dependency; easy to diff and archive |
